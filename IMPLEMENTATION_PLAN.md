# ADO Dashboard — Implementation Plan

## Project Overview
Build a platform-agnostic web dashboard that fetches work item data directly from Azure DevOps (ADO) REST API, applies lightweight data transformations, and displays visual charts. No Power BI dependency. Deployed to Azure via automated CI/CD.

## Tech Stack
- **Frontend**: Plain HTML + CSS + vanilla JavaScript
- **Backend**: Python FastAPI (Azure Functions with custom handler)
- **Hosting**: Azure Static Web Apps (frontend) + Azure Functions (API)
- **CI/CD**: GitHub Actions (auto-generated by Azure Static Web Apps on repo link)
- **Data Source**: Azure DevOps REST API / Analytics OData API
- **Charts**: Chart.js (CDN, no build step needed)
- **Auth**: None for prototype (added in Phase 6)

## Phase 1: Project Setup & Automated Deployment Pipeline

### Goal
GitHub repo connected to Azure with auto-deploy on every push to main.

### Steps
1. Create the folder structure above
2. Add a `.gitignore` covering `__pycache__`, `.env`, `local.settings.json`, `venv/`
3. Add a basic `frontend/index.html` placeholder so first deploy succeeds
4. Add `staticwebapp.config.json` at repo root:
```json
{
  "routes": [
    {
      "route": "/api/*",
      "allowedRoles": ["anonymous"]
    },
    {
      "route": "/*",
      "serve": "/index.html",
      "statusCode": 200
    }
  ]
}
```
5. Push to main and confirm GitHub Actions deploys successfully

### Expected Output
Live URL (`https://<name>.azurestaticapps.net`) showing placeholder page.

## Phase 2: Azure DevOps Data Integration

### Goal
A FastAPI HTTP endpoint that authenticates with ADO, fetches work items, and returns raw JSON.

### ADO Auth
Use a Personal Access Token (PAT). Store as environment variable — never hardcode.
- Local: `api/local.settings.json` → `ADO_PAT`
- Production: Azure Static Web Apps → Configuration → Application Settings → `ADO_PAT`

### Python Dependencies (`api/requirements.txt`)
```
fastapi
httpx
azure-functions
```

### ADO Endpoints to Use

**Option A — WIQL query (work item IDs first, then batch fetch details):**
```
POST https://dev.azure.com/{organization}/{project}/_apis/wit/wiql?api-version=7.1
GET  https://dev.azure.com/{organization}/{project}/_apis/wit/workitems?ids={ids}&fields={fields}&api-version=7.1
```

**Option B — Analytics OData (better for aggregated/reporting data):**
```
GET https://analytics.dev.azure.com/{organization}/{project}/_odata/v4.0/WorkItems
```

> ⚠️ **STOP HERE** — Before writing fetch logic, confirm with user:
> 1. What is your ADO **organization name** and **project name**?
> 2. Which **work item types** to fetch? (User Story, Bug, Task, Epic, Feature — or all?)
> 3. Which **fields** per work item? (ID, Title, State, AssignedTo, IterationPath, Priority, StoryPoints, CreatedDate, ClosedDate)
> 4. Filter by **iteration/sprint**, **team**, or **area path**? Or fetch everything?
> 5. WIQL queries or Analytics OData endpoint?

### Expected Output
`GET /api/getWorkItems` returns a JSON array of raw work items from ADO.

## Phase 3: End-to-End Prototype with Live Data

### Goal
Plain HTML/CSS/JS frontend with a Refresh button that calls the FastAPI endpoint and displays raw data. Confirm full flow works in production.

### Steps
1. `frontend/index.html` — basic layout with:
   - Dashboard title
   - **Refresh** button
   - `<pre>` tag to dump raw JSON response
   - Loading indicator

4. Push to main and verify full flow on production URL

### Expected Output

## Phase 4: Data Transformation & Dashboard Visualizations

### Goal
Transform raw ADO data into meaningful metrics and display as visual charts using Chart.js.

> ⚠️ **STOP HERE** — Before writing transformation logic, confirm with user:

## Phase 5: Performance, Reliability & Security Hardening

### Goal
Make the dashboard production-ready with proper ADO auth and optional infrastructure improvements. No caching layer — keeps infrastructure free and simple.

### Caching decision
No server-side cache. Client-side filtering (implemented in Phase 4) already eliminates per-interaction latency. ADO is only hit on explicit Refresh — acceptable for this use case. Azure Table Storage and Redis both add cost and complexity with minimal benefit given the current usage pattern.

### Steps
1. Replace PAT with **Azure AD app registration** for ADO auth (more secure, no manual rotation)
2. Add **custom domain** and SSL via Azure Static Web Apps → Custom Domains

## Phase 6: User Authentication & Access Control

### Goal
Lock the dashboard to authorised users only via Azure AD login.

### Steps
1. Register an app in Azure AD, collect Client ID, Tenant ID, Client Secret
2. Update `staticwebapp.config.json`:
```json
{
  "auth": {
    "identityProviders": {
      "azureActiveDirectory": {
        "registration": {
          "openIdIssuer": "https://login.microsoftonline.com/{tenant-id}/v2.0",
          "clientIdSettingName": "AAD_CLIENT_ID",
          "clientSecretSettingName": "AAD_CLIENT_SECRET"
        }
      }
    }
  },
  "routes": [
    {
      "route": "/*",
      "allowedRoles": ["authenticated"]
    }
  ],
  "responseOverrides": {
    "401": {
      "redirect": "/.auth/login/aad",
      "statusCode": 302
    }
  }
}
```
3. Add `AAD_CLIENT_ID` and `AAD_CLIENT_SECRET` to Azure Static Web Apps → Configuration → Application Settings
4. Push and verify login wall redirects unauthenticated users to Microsoft login
